要件定義から実装まで
― 社内DB構築プロジェクト

要件定義 テーブル設計 正規化 実装 法人部門向けの「取引先サービス紹介DB」を、要望のヒアリングから設計・実装まで一貫して担当した記録です。

1 発端 ─ 部門からの要望

統括部長
営業職員が、自分の担当以外の取引先のサービスも顧客に紹介できるようにしたい。 ホテル、小売、飲食…いろんな業種の取引先があるから、それを横断で探せるDBを作れないかな。
わたし
承知しました。まずどんな情報が必要か、現場の業務担当者にヒアリングしてから、設計に入ります。

※ 統括部長の要望を私の上司が受け、上司から私に設計・実装が任されました。統括部長案件を一任される形で担当しました。

2 要件定義 ─ 何が必要かを引き出す

▸ ACTION ─ ヒアリング

業務担当者への直接ヒアリング 私が担当

「何が欲しいか」だけでなく、何のために使うのか・どう使うのかまで踏み込んで確認しました。私自身も業務側を理解していたため、ゼロから聞くのではなく認識のすり合わせを中心に効率的に進行。

  • 営業職員は「宴会・営業の場面で、他職域の取引先サービスを顧客に紹介したい」
  • 紹介して利用されると、紹介元の営業職員の実績(企業へのアピール)になる
  • 登録は1サービスずつ。更新は担当営業職員自身ができるようにしたい
  • 担当営業に直接連絡できるフォームが欲しい
▸ DELIVERABLE ─ 成果物

項目一覧と画面イメージを作成 私が担当

ヒアリング内容を、Excelで項目一覧と画面イメージに整理。上司・担当部署の部長のレビュー(赤入れ)を受けて、認識を合わせながら固めました。

▸ ISSUE ─ 一番苦労した点

業種の「分類と見せ方」の定義

業種・サービス・特典が混在し、どう整理して画面で表現するかに苦労しました。業務担当者と何度もすり合わせ、最終的に下記のような表現に統一しました。

飲食(業種) 飲食サービス ポイント10アップ(特典)

あわせて、本当に必要な情報だけに絞り、不要な入力項目を省くことで、現場の入力負担とミスを減らすことも意識しました。

3 テーブル設計 ─ 正規化でデータを分ける

なぜテーブルを分けたか

1つの法人が複数のサービスを持つ(例:ホテル → 宿泊+レストラン)。1つの表に詰め込むと「サービス1・サービス2…」と列が増え続けて破綻します。そこで、データの種類ごとにテーブルを分けました(正規化)。

corporations（法人）
corp_idPK
corp_name 会社名
category_id 業種FK
sales_id 担当営業FK
services（サービス）
service_idPK
corp_id 法人FK
service_name サービス名
benefit 特典(ポイント等)
sales_staff（営業職員）/ categories（業種マスタ）
sales_id / category_idPK
担当者名・連絡先 / 業種名

担当営業や業種を別テーブル(マスタ)にしてIDで参照。担当者変更や業種名修正が1か所の更新で済むようにしました(第3正規形の考え方)。

4 実装 ─ クラス設計と共通化

テーブルに対応するクラスを設計

扱うデータごとにクラスを分け、テーブル構造と対応させました。

  • 法人クラス / サービスクラス / 営業職員クラス ― 各データと、その取得・更新処理を保持
  • 法人1社にサービスが複数紐づくので、サービス側から法人を参照する形に

共通処理をまとめる（汎用化）工夫した点

どのデータも「DBから検索する・登録する・更新する」という共通の処理があったため、その部分を共通のDB操作クラスにまとめ、各クラスがそれを使う形にしました。

// 共通のDB操作をまとめた親クラス（イメージ）
class BaseDao {
    findById(id)  { /* 共通の検索処理 */ }
    insert(data) { /* 共通の登録処理 */ }
    update(data) { /* 共通の更新処理 */ }
}
// 各クラスは共通処理を引き継いで使う
class CorporationDao extends BaseDao { ... }
class ServiceDao     extends BaseDao { ... }

当初は同じような処理を繰り返し書いていましたが、上司から「繰り返す処理は共通化するように」と指導を受け、保守性を意識した設計に改めました。修正が必要なときも1か所直せば済みます。

5 結果 ─ 現場で使われ続けた

2年以上、現場で活用

2年+ 継続して使用された
  • 統括部長が部門内に活用を呼びかけ、定着した
  • 運用後、機能への不満の声は出なかった
  • 「現場に本当に必要とされるものを作れた」ことが一番の手応え

このプロジェクトでできたこと

  • 要件定義:現場ヒアリングから必要な情報を引き出し、項目・画面イメージに落とせる
  • 業務理解:業務側の視点で「使う人の負担」まで考えた設計ができる
  • テーブル設計・正規化:データの構造に応じて適切にテーブルを分けられる
  • 実装・保守性:共通化・汎用性を意識したクラス設計
  • 一気通貫:上流から運用まで、一連の流れを実務で経験
