安全を意識したUPDATE業務

DML SELECT → UPDATE 本番運用の基本動作
上司
法人部門から依頼がきてる。ある業務ID（情報通信料）の案件を出している法人の名前の先頭に、識別用の文字をつけてほしいって。
業務ID 25 の法人名の頭に識別用の『N』をつけておいてくれる？
わたし
承知しました。まず何件が対象になるか確認してから更新しますね。

▸ 業務の進め方

!
いきなり UPDATE で更新しない。本番データを壊すリスクがある。
1
まず SELECT で対象を一覧表示。想定した法人が対象になっているか、件数が想定どおりかを確認する。
2
確認できたら UPDATE を実行して、法人名の先頭に「N」を付加する。
3
更新後も SELECT で確認。正しく反映されたか、Before / After を目視チェックする。
⏳ データベースを初期化しています...

▸ STEP 1 ─ SELECTで対象を確認する

-- まず、どの行が影響を受けるか確認する
SELECT id, business_id, corp_name, rep_name
FROM corporation
WHERE business_id = 25;

▸ STEP 2 ─ UPDATEを実行する

-- トランザクション開始
-- 更新前に対象確認済み

BEGIN TRANSACTION;

UPDATE corporation
SET corp_name = 'N' || corp_name
WHERE business_id = 25
AND corp_name NOT LIKE 'N%';

-- 確認後に確定
COMMIT;

-- 問題があれば取り消し
-- ROLLBACK;

▸ 解説

なぜSELECTを先に実行するのか

  • 件数の確認:「4件のはずが400件ヒット」→ WHERE句の間違いに気づける。
  • 内容の確認:対象外のデータが混ざっていないか目視チェックできる。
  • 全件更新の防止:WHERE句を書き忘れた場合、SELECTなら全件表示されるだけで済む。UPDATEだと全件書き換わって大事故になる。
  • トランザクション運用:実務では BEGIN → UPDATE → 結果確認 → COMMIT or ROLLBACK というフローになる。

▸ メモ

「本番環境でUPDATE文を実行するとき、まずSELECTで同じWHERE句の結果を確認してから実行するようにしていました。件数が想定と違えばWHERE句の間違いに気づけますし、万が一の全件更新事故を防げます。このデモでは業務ID25の法人名を更新する手順を再現しています。」
